# 需求交付引擎功能设计

## 1. 文档目标

本文档作为“AI 驱动需求交付流程引擎”的实现基线，覆盖以下范围：

- 角色与权限模型
- 核心业务流程与状态流转
- 服务端模块划分与职责
- 数据模型与持久化方案
- REST API 契约
- Eino 工作流设计
- 飞书 OAuth、消息、文档、多维表格集成方案
- 前端工作台界面与交互约束
- 异步任务、审计、失败重试和扩展策略

本文档对应当前仓库中的以下落地点：

- 前端：`apps/web`
- 后端：`apps/api-go`
- 共享类型：`packages/shared`
- 飞书应用配置：`apps/app.manifest.json`

## 2. 产品目标

系统目标是让产品经理在需求会话中与 AI 反复打磨需求，确认后通过“发布需求”动作触发确定性工作流，自动完成需求拆解、负责人匹配、任务文档生成、飞书消息分发和多维表格写入，并在后续交付周期中持续同步状态。

核心设计原则：

- 聊天不等于正式需求，必须经过显式发布。
- 权限基于“身份同步 + 系统角色映射 + 负责人快照”，不直接依赖当前部门名称。
- Agent 工作流按节点拆分，可观测、可重试、可审计。
- 长文本沉淀在文档，表格只承载排期与状态。
- 飞书接口统一封装，业务层不直接拼接裸请求。
- 原型使用 SQLite，但从代码结构上抽象仓储层，后续可切换 PostgreSQL。

## 3. 角色与权限

### 3.1 系统角色

- `product`：创建需求、编辑草稿、发布需求、查看全部任务。
- `frontend`：查看自己参与的需求与前端任务，持续跟进状态。
- `backend`：查看自己参与的需求与后端任务，持续跟进状态。
- `admin`：维护角色映射、知识源、默认排期和飞书配置。

### 3.2 权限判定规则

1. 用户通过飞书 OAuth 登录。
2. 后端拉取用户基础资料、职位、部门列表。
3. 系统根据角色映射规则计算系统角色。
4. 需求发布时，把负责人写入任务和需求快照。
5. 后续查询历史需求时，以快照和任务归属为准，不受组织变动影响。

### 3.3 角色映射来源

- 飞书部门名称关键字
- 职位关键字
- 人工覆盖配置

## 4. 关键流程

### 4.1 登录与身份同步

1. 前端跳转 `GET /api/auth/feishu/login`
2. 后端生成 state 并重定向到飞书 OAuth
3. 飞书回调 `GET /api/auth/feishu/callback`
4. 后端换取用户身份、同步部门和系统角色
5. 写入本地 session cookie
6. 前端请求 `GET /api/me` 获取当前用户

### 4.2 草稿需求对话

1. 产品创建会话 `POST /api/sessions`
2. 首条消息作为需求初稿写入会话
3. 后端用轻量规则摘要生成一条 AI 回复
4. 会话保持 `draft` 状态，允许继续对话

### 4.3 发布需求

1. 产品点击“发布需求”
2. 后端校验角色、会话状态和消息内容
3. 生成 `requirement` 快照并将会话状态改为 `published`
4. 投递后台作业
5. 后台依次执行：
   - 需求归一化
   - RAG 检索
   - 任务拆解
   - 负责人匹配
   - 文档生成
   - 飞书消息发送
   - 多维表格写入
   - 结果持久化
6. 执行成功后状态进入 `in_delivery`

### 4.4 研发跟进与状态同步

1. 研发查看自己的任务列表
2. 更新任务状态 `PATCH /api/tasks/{taskID}/status`
3. 后端写库并回写飞书多维表格
4. 当全部任务到达 `testing` 或 `done` 时，可进一步推进需求会话状态

## 5. 状态模型

### 5.1 会话状态

- `draft`
- `published`
- `in_delivery`
- `testing`
- `done`
- `archived`

### 5.2 任务状态

- `todo`
- `in_progress`
- `testing`
- `done`

### 5.3 状态推进约束

- 只有 `draft` 会话允许发布。
- 发布后仍允许继续聊天，但不会隐式覆盖正式拆解结果。
- 任务状态变化可触发对应的会话状态聚合更新。

## 8. API 设计

所有响应统一为：

```json
{
  "data": {}
}
```

失败时：

```json
{
  "error": "error message"
}
```

### 8.1 健康检查

- `GET /api/health`

### 8.2 认证

- `GET /api/auth/feishu/login`
- `GET /api/auth/feishu/callback?code=...&state=...`
- `GET /api/me`

### 8.3 会话

- `GET /api/sessions`
- `POST /api/sessions`
- `GET /api/sessions/{sessionID}`
- `POST /api/sessions/{sessionID}/messages`
- `POST /api/sessions/{sessionID}/publish`

### 8.4 任务

- `GET /api/tasks/{taskID}`
- `PATCH /api/tasks/{taskID}/status`

### 8.5 管理接口

- `POST /api/admin/role-mappings`
- `POST /api/admin/knowledge/sync`

## 9. Eino 工作流

### 9.1 工作流节点

- `normalize`
  - 输入：会话标题、聊天历史
  - 输出：结构化需求摘要

- `rag`
  - 输入：归一化后的需求摘要
  - 输出：命中的知识片段列表

- `split`
  - 输入：需求摘要 + 知识片段
  - 输出：前端 / 后端 / 公共任务

- `assign`
  - 输入：任务列表 + 角色映射
  - 输出：带负责人的任务列表

- `docs`
  - 输入：任务列表
  - 输出：Markdown 文档内容

- `delivery`
  - 输入：任务文档
  - 输出：飞书消息、文档、多维表格计划

- `summary`
  - 输入：全部节点结果
  - 输出：发布回执摘要

### 9.2 一期实现策略

一期使用 `Eino compose.Workflow` 和 Lambda 节点实现可运行工作流：

- 结构采用 Eino 的 `Workflow`
- 每个节点内部先用确定性规则实现
- 保持接口稳定，后续可替换为真实大模型、向量检索和智能分配策略

## 10. 飞书集成设计

### 10.1 OAuth

- 配置项：
  - `FEISHU_APP_ID`
  - `FEISHU_APP_SECRET`
  - `FEISHU_REDIRECT_URL`

- 无飞书配置时，后端进入本地开发降级模式：
  - 直接写入演示用户 cookie
  - 页面仍可完整体验主链路

### 10.2 消息

- 每个任务至少发送一条提醒消息
- 记录消息投递结果，便于重试和审计

### 10.3 文档

- 一期以 Markdown 内容和模拟文档 URL 为主
- 配置飞书后可调用云文档接口创建真实文档

### 10.4 多维表格

- 每个任务对应一条记录
- 字段：
  - 需求标题
  - 子任务类型
  - 负责人
  - 状态
  - 文档链接
  - 最近更新时间

## 11. 异步作业与失败重试

- 发布需求不阻塞前端
- 后端用单 worker 队列消费发布任务
- 节点失败时写错误日志和会话摘要
- 可通过再次发布或专门的重试入口继续执行

## 12. 前端页面设计

### 12.1 登录头部

- 展示当前登录用户
- 飞书登录按钮
- 发布需求按钮

### 12.2 会话列表

- 展示标题、状态、更新时间、消息数
- 支持新建会话

### 12.3 聊天区

- 展示消息时间线
- 输入需求描述或补充问题
- 发送后即时刷新

### 12.4 详情侧栏

- 展示正式摘要
- 展示任务列表
- 展示引用知识片段
- 支持修改任务状态

## 13. 安全与治理

- 所有写接口基于本地 session 用户
- 管理接口限制为 `admin`
- 飞书 token 仅存放服务端
- Agent 输出进入数据库前必须结构化校验
- 对外返回统一 JSON，避免暴露内部错误堆栈
