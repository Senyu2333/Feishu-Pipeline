# 飞书智交付 · 产品开发设计

## 1. 文档定位

本文档用于将当前仓库中的 Demo 方案升级为一套能够对齐《飞书智交付赛题》要求的完整产品与技术设计。目标不是只描述一个“需求拆解工具”，而是定义一套 **AI 驱动的研发流程引擎（DevFlow Engine）**，使平台能够从自然语言需求出发，经过需求分析、方案设计、代码生成、测试生成、代码评审、交付集成等阶段，最终产出可运行、可评审、可合并的代码变更。

本文档同时覆盖赛题功能二的理想形态：在浏览器页面中注入 Agent 浮层，支持圈选元素、自然语言修改、热更新预览与 MR 提交，形成“页面即工作台”的 AI Native 研发闭环。

本文档既服务于后续研发实现，也服务于比赛答辩、Demo 演示和项目交付材料准备。

---

## 2. 赛题目标拆解与产品定位

### 2.1 赛题要求拆解

结合《飞书智交付赛题》，平台必须满足以下能力：

1. **Pipeline 引擎能力**
   - 支持 Stage 定义、依赖编排、输入输出流转
   - 支持启动、暂停、恢复、终止、重跑
   - 每个 Stage 可以绑定一个或多个 Agent

2. **Agent 编排能力**
   - 每个阶段有明确的 Agent 角色、Prompt、输入输出契约
   - Agent 能感知代码库上下文
   - LLM Provider 可切换，至少支持 2 个模型提供方

3. **Human-in-the-Loop 能力**
   - 至少 2 个人工检查点
   - 支持 Approve / Reject / 退回重做
   - 有明确 UI 或 API 展示阶段产出

4. **API First 架构**
   - Pipeline CRUD、执行触发、状态查询、检查点操作必须 API 化
   - 提供 OpenAPI/Swagger 文档

5. **端到端交付能力**
   - 从需求输入到可交付代码变更的完整闭环
   - 可将目标仓库设置为本平台自身，实现自举演示

6. **浏览器在页改动能力（功能二）**
   - 页面注入浮层
   - 元素圈选
   - 对话驱动修改
   - 热更新预览
   - 自动创建 MR/PR 并生成摘要

### 2.2 产品定位

本项目定位为：

> 一套面向研发团队的 AI Native DevFlow 平台，让 AI 成为研发流水线的主驾驶，人类只在关键节点进行决策与验收。

平台目标用户不只包括开发者，也包括产品经理、设计师、测试与技术负责人：

- 产品经理：输入需求、查看结构化分析、审批方案
- 技术负责人：审核设计、控制风险、审批代码评审结果
- 开发者：查看 Agent 生成的代码、执行本地验证、处理少量人工修正
- 测试/QA：查看测试结果与验收报告
- 设计/运营：直接在页面上圈选元素，通过对话驱动修改并实时预览

### 2.3 当前 Demo 与目标形态的关系

当前仓库已经具备以下基础：

- Web 前端 `apps/web`
- Go 后端 `apps/api-go`
- 会话、任务、发布、飞书登录等基础能力
- 一套初步的 Agent / Workflow / Pipeline 控制器结构

但当前方案仍偏向“需求对话 + 拆任务 + 同步飞书”的 Demo，不足以完整覆盖赛题要求。后续设计将以 **保留现有代码骨架、升级为完整 DevFlow Engine** 为原则，避免推倒重来。

## 3. 产品愿景与核心原则

### 3.1 产品愿景

让研发流程从“人工串联”升级为“AI 编排 + 人工监督”的新范式：

- 需求由 AI 转换为结构化产物
- 方案由 AI 结合代码库自动生成
- 编码与测试由 AI 执行
- 评审由 AI 先做第一轮质量门禁
- 人类只在高价值决策点介入

### 3.2 核心原则

1. **Pipeline 是骨架，Agent 是执行单元**
   - 平台关注阶段编排、状态、可追踪性
   - Agent 关注阶段内的专业任务完成

2. **先结构化，再执行**
   - 任何自然语言输入，都必须先转化为结构化需求/结构化任务/结构化方案，避免信息在流程中失真

3. **所有关键节点可观测、可重放、可审计**
   - 每个 Stage 的输入、输出、日志、耗时、token、错误都可追踪

4. **人类只做高价值决策**
   - 人工介入点只放在风险最高或需要业务判断的阶段

5. **先实现可演示闭环，再逐步增强智能度**
   - 一期优先保证端到端可运行
   - 二期增强多 Agent 协作、自动修复、语义检索、在页编辑体验

6. **API First + UI 驱动**
   - 所有能力通过 API 暴露
   - 前端只是工作台，不承载核心业务状态机

## 4. 总体产品架构

### 4.1 逻辑架构

系统分为六层：

1. **交互层**
   - Web 控制台（需求输入、Pipeline 工作台、审批工作台）
   - 浏览器扩展 / 页面注入浮层（在页圈选与编辑）
   - 飞书消息入口（消息通知、审批提醒）

2. **API 网关层**
   - REST API
   - 鉴权、会话、权限校验
   - OpenAPI 文档

3. **Pipeline 引擎层**
   - Pipeline 模板管理
   - Stage 编排与依赖控制
   - 生命周期管理
   - Checkpoint 挂起与恢复
   - 重试、回滚、补偿

4. **Agent 编排层**
   - Agent Registry
   - Prompt 模板
   - Context Builder
   - Tool 调用封装
   - 多 Provider 路由

5. **执行与集成层**
   - Git 工作目录管理
   - 代码检索与语义索引
   - 命令执行器（测试、构建、lint）
   - 飞书集成
   - Git 平台集成（GitHub/GitLab）
   - 热更新预览协调器

6. **数据与可观测层**
   - PostgreSQL / SQLite
   - 对象存储（阶段产物、日志、diff）
   - 向量索引（可选）
   - 事件日志 / 审计日志
   - 指标与追踪

### 4.2 建议技术栈

#### 前端

- **React 18 + TypeScript**：构建 SPA 工作台
- **Vite**：本地开发与热更新速度快，适合赛题演示
- **React Router**：多页面工作台路由
- **TanStack Query**：服务端状态管理，适合任务、流水线、审批轮询
- **Zustand**：轻量本地状态，适合页面浮层、圈选状态、聊天上下文
- **Tailwind CSS**：快速构建飞书风格工作台界面
- **Monaco Editor / Diff Editor**：展示生成代码、diff、审阅结果
- **SSE/WebSocket**：阶段状态流式更新

#### 后端

- **Go 1.24**：作为主后端，适合实现 Pipeline 引擎、并发调度和飞书/Git/LLM 集成
- **Gin / Chi（二选一，建议 Gin）**：REST API 框架
- **GORM / sqlc（二选一，建议 GORM 先行）**：加快 Demo 落地，后续可优化
- **Swagger / OpenAPI 3**：规范输出 API 文档
- **Viper**：配置管理
- **Zap**：结构化日志
- **Robfig/cron + 自研 Job Runner**：调度后台任务

#### AI 与 Agent

- **Eino Workflow**：继续沿用当前方向，用于实现 Agent Workflow 编排
- **Anthropic Claude / OpenAI / 火山 / 通义（至少两家）**：通过 Provider Adapter 抽象统一接口
- **Embedding + 向量检索（pgvector / Qdrant）**：用于代码库与知识库语义召回
- **JSON Schema Output**：强约束 Agent 输出结构，降低幻觉对流程的破坏

#### 数据与基础设施

- **SQLite**：Demo 默认本地运行数据库
- **PostgreSQL**：正式版推荐数据库，支持 JSONB、事务、扩展性更强
- **Redis（可选）**：作业队列、运行时缓存、checkpoint 恢复信号
- **Docker Compose**：比赛交付的一键启动方案
- **Nginx**：前端静态资源与反向代理

#### 集成能力

- **飞书 OAuth / 消息卡片 / 云文档 / 多维表格**
- **GitHub/GitLab API**：分支、提交、PR/MR 创建
- **本地 Git CLI**：在工作目录执行 checkout、diff、commit、test

## 6. Pipeline 引擎设计

## 6.1 Pipeline 核心概念

### 6.1.1 PipelineTemplate

描述某类流程的静态定义，例如：

- `feature-delivery`：新功能开发流程
- `bugfix-delivery`：缺陷修复流程
- `ui-polish-delivery`：页面微调流程
- `self-hosted-evolution`：平台自举迭代流程

字段：

- `id`
- `name`
- `description`
- `version`
- `stages[]`
- `defaultProvider`
- `variablesSchema`
- `checkpointPolicy`

### 6.1.2 StageDefinition

用于描述单个阶段：

- `key`：阶段标识
- `name`：阶段名称
- `type`：analysis/design/codegen/test/review/checkpoint/delivery
- `dependsOn[]`：依赖阶段
- `agents[]`：绑定的 Agent 定义
- `inputMapping`
- `outputSchema`
- `retryPolicy`
- `timeoutSeconds`
- `humanCheckpointConfig`

### 6.1.3 PipelineRun

某次真实执行实例，包含：

- `runId`
- `templateId`
- `targetRepo`
- `targetBranch`
- `status`
- `currentStage`
- `contextSnapshot`
- `createdBy`
- `startedAt`
- `finishedAt`

### 6.1.4 StageRun

每个阶段的运行实例，包含：

- `stageRunId`
- `pipelineRunId`
- `stageKey`
- `status`
- `attempt`
- `inputPayload`
- `outputPayload`
- `errorMessage`
- `tokenUsage`
- `startedAt`
- `finishedAt`

## 6.2 Pipeline 生命周期

状态建议如下：

- `draft`：已创建，未执行
- `queued`：已入队等待执行
- `running`：执行中
- `waiting_approval`：等待人工审批
- `paused`：人工暂停
- `failed`：失败
- `completed`：已完成
- `terminated`：已终止

支持动作：

- `start`
- `pause`
- `resume`
- `terminate`
- `retry_stage`
- `rerun_from_stage`
- `approve_checkpoint`
- `reject_checkpoint`

## 6.3 Stage 依赖与调度策略

一期采用 **DAG + 顺序执行为主，局部并行为辅** 的调度策略：

- 需求分析 → 方案设计 → 代码生成 为主链顺序
- 测试生成可与代码评审并行准备
- 安全审查、规范审查可并行跑多个 Review Agent
- 交付集成依赖所有门禁通过

调度器逻辑：

1. 获取所有未完成 Stage
2. 筛选依赖已满足的 Stage
3. 判断是否达到并发上限
4. 分配给对应 Agent Executor
5. 完成后写 StageRun 并触发下游阶段

## 6.4 Checkpoint 设计

至少包含两个强制人工检查点：

### Checkpoint A：方案设计审批

输入：

- 结构化需求
- 技术方案
- 影响文件清单
- 风险说明

审批操作：

- `approve`
- `reject`
- `reject_with_comment`

Reject 后行为：

- 将 reject reason 注入到设计 Agent 的下一次输入中
- 重新执行方案设计阶段
- 记录版本差异

### Checkpoint B：代码评审确认

输入：

- 代码 diff
- 测试结果
- AI 评审报告
- 风险点清单

Reject 后行为：

- 将问题列表传给修复 Agent
- 触发自动修复阶段
- 修复后重新走测试与评审

### 可选 Checkpoint C：发布前确认

用于控制是否自动创建 PR/MR 或发版工单。

## 7. Agent 编排设计

## 7.1 Agent 角色定义

平台按阶段拆分 Agent，而不是让一个超级 Agent 独自完成所有事情。建议至少具备以下 Agent：

1. **Requirement Analyst Agent**
   - 输入：自然语言需求、历史对话、业务上下文
   - 输出：结构化需求、验收标准、边界条件、疑问项

2. **Solution Architect Agent**
   - 输入：结构化需求、代码库上下文、知识库结果
   - 输出：技术方案、影响范围、改动清单、接口设计

3. **Code Generation Agent**
   - 输入：技术方案、目标文件上下文、编码规范
   - 输出：逐文件修改提案或统一 diff

4. **Test Engineer Agent**
   - 输入：需求、方案、代码 diff
   - 输出：单测、集成测试、测试说明

5. **Reviewer Agent**
   - 输入：代码 diff、方案、测试结果
   - 输出：问题列表、风险等级、修复建议

6. **Fix Agent**
   - 输入：Review 问题列表、失败测试、原代码 diff
   - 输出：修复后的代码变更

7. **Delivery Agent**
   - 输入：已通过门禁的变更集
   - 输出：commit message、PR/MR 摘要、交付总结

8. **In-Page Edit Agent（功能二）**
   - 输入：DOM 选区信息、用户指令、源码映射上下文
   - 输出：UI 局部改动 diff、预览反馈

## 7.2 Agent 输入输出契约

每个 Agent 必须以 **强约束 JSON Schema** 作为输出规范，避免直接让下游消费自由文本。

例如 Requirement Analyst Agent 输出：

```json
{
  "title": "新增首页英雄区优化",
  "summary": "优化首页首屏表达，提升转化效率",
  "goals": ["提升首屏信息清晰度"],
  "nonGoals": ["不修改登录流程"],
  "acceptanceCriteria": [
    "标题文案更新为新的品牌主张",
    "CTA 按钮颜色调整为品牌蓝"
  ],
  "risks": ["可能影响首屏布局"],
  "openQuestions": []
}
```

## 7.3 Prompt 模板策略

Prompt 分为三层：

1. **System Prompt**
   - 定义 Agent 角色、边界、输出格式、禁止事项

2. **Task Prompt**
   - 注入当前阶段目标、输入数据、验收要求

3. **Context Prompt**
   - 注入代码片段、目录树、接口定义、历史 reject reason

Prompt 设计要求：

- 明确输出 JSON Schema
- 明确禁止编造不存在的文件/接口
- 明确优先复用现有代码结构
- 明确安全约束（不允许引入危险命令执行等）
- 明确遇到信息不足时返回 `uncertainties`

## 7.4 多 Provider 模型路由

为了满足赛题要求并兼顾成本/质量，平台应支持至少两个 Provider：

- Provider A：Anthropic Claude（复杂设计、审查、长上下文）
- Provider B：OpenAI / 通义 / DeepSeek（二选一，用于成本优化或对照）

统一 Provider 接口：

```go
type LLMProvider interface {
    Generate(ctx context.Context, req GenerateRequest) (*GenerateResponse, error)
    Stream(ctx context.Context, req GenerateRequest) (<-chan StreamChunk, error)
    Embedding(ctx context.Context, req EmbeddingRequest) (*EmbeddingResponse, error)
}
```

路由策略：

- 需求分析、方案设计、代码评审优先使用强推理模型
- 测试生成、简单改文案可使用成本更低模型
- 用户可在 Pipeline 模板中配置默认 Provider

## 7.5 工具调用能力

Agent 可以使用如下工具，但必须通过平台控制层统一暴露：

- 代码库搜索工具（grep / AST query）
- 文件读取工具
- 文件写入工具
- Git diff 工具
- 测试执行工具
- 构建执行工具
- 飞书文档写入工具
- PR/MR 创建工具

这样做的好处是：

- 工具权限集中控制
- 调用链可审计
- 便于在比赛答辩中解释 Agent 的能力边界

## 8. 代码库上下文与语义检索设计

## 8.1 为什么必须有代码库上下文层

赛题明确要求 Agent 能感知代码库上下文。若仅把整个仓库文本喂给模型，会导致：

- 上下文过长，成本高
- 召回不精准
- 修改位置错误率高

因此必须实现 **Repository Context Builder**。

## 8.2 上下文构建策略

分三层构建：

1. **结构层**
   - 仓库目录树
   - package/module 信息
   - 路由、组件、控制器清单

2. **语义层**
   - 文件摘要
   - 函数/组件摘要
   - 接口/类型摘要
   - 向量索引召回

3. **任务层**
   - 当前需求涉及的关键文件
   - 最近改动历史
   - reject 历史与修复记录

## 8.3 代码索引实现方案

### 一期可运行方案

- 启动时扫描目录
- 建立文件元数据表
- 通过关键字 + 路径规则召回文件
- 对重点文件做摘要缓存

### 二期增强方案

- 代码切块（按文件、函数、组件、接口）
- 使用 embedding 写入 pgvector / Qdrant
- 按需求语义召回最相关片段
- 结合 rerank 提升精度

## 8.4 推荐上下文注入内容

对于一次代码生成请求，注入：

- 目标文件完整内容（最多 2~4 个）
- 相关文件摘要（最多 5~8 个）
- 项目约定（技术栈、命名、UI 规范）
- 当前方案中声明的变更清单
- 上次失败测试和 review 问题

## 9. 页面内圈选修改设计（功能二）

## 9.1 架构设计

功能二建议拆成三个部分：

1. **浏览器扩展 / 注入脚本层**
   - 负责在页面中插入浮层 UI
   - 负责捕获用户圈选行为
   - 负责提取 DOM 元素上下文

2. **前端工作台桥接层**
   - 将圈选信息发送给后端
   - 订阅编辑执行状态
   - 控制预览确认与撤销

3. **后端 In-Page Edit Pipeline**
   - DOM → 源码定位
   - 源码修改
   - 热更新触发
   - 预览结果反馈
   - PR/MR 创建

## 9.2 浮层交互设计

浮层应包含：

- 开关按钮
- 圈选模式按钮
- 当前选中元素列表
- 自然语言输入框
- “开始修改”按钮
- “撤销修改”按钮
- “确认并创建 MR”按钮

## 9.3 元素圈选实现细节

用户进入圈选模式后：

1. 鼠标 hover 页面元素
2. 注入层为元素绘制高亮边框
3. 点击后记录目标元素信息
4. 支持多选，最少 3 个目标元素

每个选区需要采集：

- `tagName`
- `textContent`
- `classList`
- `id`
- `data-*`
- XPath / CSS Selector
- bounding box
- 父子节点路径
- 截图片段（可选）

示例：

```json
{
  "selector": "button.hero-cta",
  "text": "立即开始",
  "componentHints": ["HeroSection", "Button"],
  "domPath": ["div#__next", "main", "section.hero", "button.hero-cta"],
  "rect": {"x": 100, "y": 220, "width": 140, "height": 44}
}
```

## 9.4 DOM 到源码定位策略

这是功能二成败的关键。建议采用多信号融合定位：

1. **className / testid / data-attr 精确匹配**
2. **文本内容反向搜索**
3. **路由页面到组件树映射**
4. **组件源码 AST 分析**
5. **运行时 sourcemap（可选）**

定位流程：

1. 根据当前页面 URL 找到路由入口文件
2. 根据 DOM 特征召回候选组件
3. 比较组件 JSX 文本与 className
4. 输出候选文件列表与置信度
5. 当置信度不足时，先进入“定位确认”而不是直接改代码

## 9.5 热更新预览实现

前提：目标项目运行在 Vite dev server 或其他支持 HMR 的开发服务器上。

流程：

1. In-Page Edit Agent 生成代码变更
2. 后端写入工作目录文件
3. 本地 dev server 触发 HMR
4. 注入浮层监听页面刷新信号
5. 将更新结果在浮层中提示

为保证演示稳定性，建议：

- 比赛 Demo 站点必须使用 Vite + React
- 页面修改优先聚焦静态文案、样式、组件属性，不在现场演示复杂业务逻辑改动

## 9.6 在页修改的安全约束

- 不允许注入任意脚本到非白名单域名
- 只允许在自建站点域名内启用
- 对用户修改指令做基本边界控制
- 对生成代码先进行语法检查，避免直接写入不可编译内容

## 10. 人机协同设计

## 10.1 为什么必须有人类介入

完全自动化并不现实，尤其在以下环节：

- 需求边界存在歧义
- 技术方案涉及架构取舍
- 代码修改可能引入高风险副作用
- 需要决定是否发起 MR/PR

因此，平台强调的是 **Human-in-the-Loop**，不是纯无人值守。

## 10.2 介入节点与责任人

1. **需求澄清节点**：产品经理确认结构化需求
2. **方案审批节点**：技术负责人审批设计方案
3. **评审确认节点**：开发负责人确认代码评审结果
4. **交付确认节点**：产品/开发确认是否创建 PR/MR

## 10.3 审批信息展示要求

审批页必须展示：

- 当前阶段名称
- 阶段输入摘要
- 阶段输出物
- 风险提示
- Agent 推理结论摘要
- 上一轮 reject 原因（如有）

## 14. 关键执行流程设计

## 14.1 从需求到交付的主流程

### Stage 1：需求分析

输入：

- 用户自然语言需求
- 历史对话
- 业务背景

处理：

- 识别目标、范围、非目标、验收标准、风险
- 产出结构化需求 JSON 与需求摘要文档

输出：

- `structured_requirement`
- `acceptance_criteria`
- `open_questions`

### Stage 2：方案设计

输入：

- 结构化需求
- 代码库索引结果
- 相关文件摘要

处理：

- 分析现有架构
- 识别改动入口
- 输出技术实现方案、文件变更计划、接口与数据结构改动

输出：

- `solution_design`
- `change_plan`
- `impact_files`

### Stage 3：方案审批 Checkpoint

由人工决定是否允许进入编码阶段。

### Stage 4：代码生成

输入：

- 技术方案
- 关键文件内容
- 编码约束

处理：

- 生成逐文件改动
- 先写入临时工作目录
- 执行语法检查

输出：

- `code_diff`
- `modified_files`

### Stage 5：测试生成与执行

输入：

- 需求
- 代码 diff
- 项目测试框架信息

处理：

- 生成单元测试 / 集成测试
- 执行测试、构建、lint
- 收集失败信息

输出：

- `test_files`
- `test_report`
- `build_report`

### Stage 6：代码评审

输入：

- 方案
- diff
- 测试结果

处理：

- 从正确性、安全性、可维护性、规范性四个维度审查
- 输出风险等级和修复建议

输出：

- `review_report`
- `issue_list`

### Stage 7：评审确认 Checkpoint

人工确认是否：

- 直接进入交付
- 触发自动修复后再审

### Stage 8：交付集成

输入：

- 最终 diff
- 测试与评审通过状态

处理：

- 创建分支
- 提交代码
- 创建 PR/MR
- 生成改动摘要与行级 diff 摘要

输出：

- `delivery_summary`
- `pr_url`
- `commit_sha`

## 14.2 自动修复闭环

当 Review Agent 或测试阶段发现问题时：

1. 生成问题清单
2. 传递给 Fix Agent
3. Fix Agent 修改代码
4. 重新执行测试
5. 重新执行评审
6. 达到最大重试次数后转人工处理

建议最大重试次数：`2`

## 15. Git 与交付设计

## 15.1 工作目录策略

为避免污染主仓库，平台执行时应采用独立工作目录：

- 基于目标仓库创建工作副本
- 创建 `devflow/<run-id>` 分支
- 所有 Agent 修改都落在该分支

对于平台自举演示，可以直接对当前仓库执行，但仍推荐：

- 使用独立工作目录或临时分支
- 保留主分支干净

## 15.2 Git 执行步骤

1. clone / checkout 仓库
2. 创建工作分支
3. 应用代码变更
4. 执行 lint / test / build
5. 生成 commit message
6. 创建 commit
7. 推送远程
8. 创建 PR/MR

## 15.3 PR/MR 摘要生成要求

PR/MR 描述应包含：

- 需求背景
- 本次改动摘要
- 关键文件变更列表
- 测试结果
- 风险提示

示例：

```md
## Summary
- 优化首页 Hero 区主标题与副标题表达
- 更新主 CTA 按钮样式与 hover 状态
- 新增次级说明卡片区块并完成样式适配

## Changed Files
- apps/web/src/App.tsx
- apps/web/src/index.css

## Validation
- pnpm --filter web build
- 页面热更新预览已验证
```

## 19. 安全设计

1. 所有 Provider Key 仅存服务端
2. Git Token、飞书 Token 仅服务端使用
3. Agent 输出必须结构化校验
4. 文件修改必须限制在白名单仓库目录中
5. 命令执行器只允许执行白名单命令（如测试、构建、lint）
6. 在页注入能力仅允许在白名单域名启用
7. 对审批、创建 PR/MR 等动作做角色校验
8. 对外接口不暴露内部错误堆栈

## 23. 比赛演示方案建议

## 23.1 环节一：平台能力展示

建议准备一个“优化平台首页 Hero 区”的需求作为预设 Demo：

- 需求清晰

- 使用自举方式完成“平台改平台”的闭环

## 23.3 环节三：技术答辩

答辩时建议突出：

1. 为什么要把流程抽象成 Pipeline + Agent + Checkpoint
2. 为什么使用结构化输出与多阶段编排，而不是单次对话生成代码
3. 为什么设计在页修改能力，并如何解决 DOM 到源码定位问题

## 24. 最终目标形态总结

本项目的理想形态不是一个“会聊天的需求拆解工具”，而是一个完整的 **AI Native 研发操作系统**：

- 对产品经理，它能把需求自动变成结构化交付流程
- 对技术负责人，它能把方案、风险、评审和审批集中在一个平台
- 对开发者，它能自动生成、验证、修复代码并发起交付
- 对设计与运营，它能直接在页面上圈选并修改，实现所见即所得的协同开发

平台既能作为比赛作品完成端到端演示，也能作为后续持续迭代的真实产品骨架。
