# 飞书智交付 · 第一阶段实施计划

## 1. 文档目标

本文档只聚焦第一阶段实施，目标是把当前项目从“需求会话 + 发布拆解”的 Demo，升级为具备 **Pipeline 核心领域模型** 的 DevFlow Engine 基础版本。

第一阶段不追求把整条赛题链路一次做完，而是优先完成后续所有能力的底座建设，确保后面做 Pipeline 执行、人工审批、前端工作台、在页编辑时，不需要再次推翻基础结构。

本文档作为第一阶段的执行基线。后续我对项目进行改造时，将默认参考本文档。

---

## 2. 第一阶段目标范围

### 2.1 本阶段目标

本阶段要解决的问题是：

1. 当前系统主模型还是 `Session / Requirement / Task`
2. 赛题真正需要的主模型应该是 `PipelineTemplate / PipelineRun / StageRun / Artifact / Checkpoint`
3. 如果不先补齐这些模型，后续所有 API、前端工作台、审批流、交付流都会变成临时拼接

因此第一阶段目标是：

- 建立后端 Pipeline 核心领域模型
- 建立持久化与状态流转能力
- 建立最小可用的 Pipeline Run API
- 前端完成对新 Pipeline 基础数据的接入准备
- 保留现有 Session 链路作为需求输入入口，不中断已有 Demo

### 2.2 本阶段不做的事情

第一阶段明确暂不做：

- 不实现完整的多阶段 Agent 执行引擎
- 不实现真实代码生成、测试执行、PR/MR 提交
- 不实现功能二的页面圈选改代码能力
- 不做多 Provider 路由细化
- 不做完整可观测性大盘

这些能力属于第二阶段及以后。

---

## 3. 第一阶段交付物

本阶段最终应交付：

### 后端交付物

1. 新的 Pipeline 核心数据模型
2. 对应 SQLite/GORM 自动迁移
3. Pipeline 相关 Repository 方法
4. Pipeline Service 基础能力
5. Pipeline Controller 基础接口
6. 路由注册
7. Pipeline 初始模板/阶段定义常量
8. 从 Session 创建 PipelineRun 的桥接能力

### 前端交付物

1. Pipeline 基础 API 调用层
2. Pipeline 列表 / 详情最小数据结构定义
3. 首页或现有入口支持创建 PipelineRun
4. 工作流页支持展示真实 PipelineRun 基础信息
5. 为后续审批页、交付页接入留出统一数据入口

## 4. 后端实施计划

## 4.1 后端阶段目标

后端第一阶段的核心是：**先把数据模型和 API 骨架搭起来**。

实现完成后，后端应具备：

- 创建 PipelineTemplate
- 基于 Session 或直接需求文本创建 PipelineRun
- 查询 PipelineRun 列表和详情
- 查询 StageRun 列表
- 查询 Artifact 列表
- 为后续阶段执行引擎预留状态流转接口

## 4.2 后端具体任务拆分

   - `Title`
   - `RequirementText`
   - `SourceSessionID`
   - `TargetRepo`
   - `TargetBranch`
   - `WorkBranch`
   - `Status`
   - `CurrentStageKey`
   - `CreatedBy`
   - `StartedAt`
   - `FinishedAt`

3. `StageRun`
   - `ID`
   - `PipelineRunID`
   - `StageKey`
   - `StageType`
   - `Status`
   - `Attempt`
   - `InputJSON`
   - `OutputJSON`
   - `ErrorMessage`
   - `StartedAt`
   - `FinishedAt`

4. `Artifact`
   - `ID`
   - `PipelineRunID`
   - `StageRunID`
   - `ArtifactType`
   - `Title`
   - `ContentText`
   - `ContentJSON`
   - `FilePath`
   - `MetaJSON`

5. `Checkpoint`
   - `ID`
   - `PipelineRunID`
   - `StageRunID`
   - `CheckpointType`
   - `Status`
   - `ApproverID`
   - `Decision`
   - `Comment`
   - `DecidedAt`

6. `AgentRun`
   - `ID`
   - `PipelineRunID`
   - `StageRunID`
   - `AgentKey`
   - `Provider`
   - `Model`
   - `PromptSnapshot`
   - `InputJSON`
   - `OutputJSON`
   - `TokenUsageJSON`
   - `LatencyMS`
   - `Status`
   - `ErrorMessage`

本任务要求：

- 为上述模型补齐枚举类型
- 定义 Pipeline/Stage/Checkpoint 状态常量
- 保持现有 `Session`、`Requirement`、`Task` 不删除
- 让老模型与新模型能并存

### 任务 B2：补 Repository 能力

修改文件：

- `apps/api-go/internal/repo/repository.go`

需要新增的方法：

1. Template 相关
   - `ListPipelineTemplates`
   - `GetPipelineTemplateByID`
   - `SavePipelineTemplate`
   - `SeedPipelineTemplates`

2. Run 相关
   - `CreatePipelineRun`
   - `GetPipelineRunByID`
   - `ListPipelineRuns`
   - `UpdatePipelineRunStatus`
   - `UpdatePipelineRunCurrentStage`

3. Stage 相关
   - `CreateStageRuns`
   - `ListStageRunsByPipelineRunID`
   - `UpdateStageRunStatus`
   - `SaveStageRunOutput`

4. Artifact 相关
   - `CreateArtifact`
   - `ListArtifactsByPipelineRunID`

5. Checkpoint 相关
   - `CreateCheckpoint`
   - `GetCheckpointByID`
   - `UpdateCheckpointDecision`

本任务要求：

- AutoMigrate 接入全部新模型
- Seed 阶段插入一个默认模板，例如 `feature-delivery`
- Repository 方法命名和现有风格保持一致

### 任务 B3：定义 Pipeline 模板与阶段常量

建议新增目录：

- `apps/api-go/internal/pipeline`

建议新增文件：

- `template.go`
- `stage.go`

内容要求：

1. 定义默认模板 `feature-delivery`
2. 定义第一版阶段清单：
   - `requirement_analysis`
   - `solution_design`
   - `checkpoint_design`
   - `code_generation`
   - `test_generation`
   - `code_review`
   - `checkpoint_review`
   - `delivery`
3. 定义阶段显示名、阶段类型、默认顺序
4. 先用静态定义，不做可视化编排

### 任务 B4：建立 Pipeline Service 基础能力

修改文件：

- `apps/api-go/internal/service/pipeline_service.go`

当前这个文件主要是飞书 bitable 逻辑，需要重构职责。

第一阶段应调整为：

1. 保留原 bitable 相关逻辑，但下沉为附属能力，不再作为主职责
2. 新增 PipelineRun 服务能力：
   - `CreatePipelineRunFromSession`
   - `CreatePipelineRun`
   - `ListPipelineRuns`
   - `GetPipelineRunDetail`
3. 创建 Run 时自动：
   - 绑定模板
   - 初始化状态为 `draft` 或 `queued`
   - 生成对应的 `StageRun` 骨架记录
4. 详情接口需要返回：
   - run 基本信息
   - stage 列表
   - artifact 列表
   - checkpoint 列表（如已有）

### 任务 B5：建立 Pipeline Controller 基础接口

修改文件：

- `apps/api-go/internal/controller/pipeline_controller.go`

第一阶段要支持的 API：

1. `GET /api/pipeline-runs`
2. `POST /api/pipeline-runs`
3. `GET /api/pipeline-runs/:id`
4. `POST /api/pipeline-runs/from-session`

其中：

- `from-session` 用来桥接现有 Session 能力
- `POST /api/pipeline-runs` 支持直接传 `title + requirementText + templateId`

返回体要求：

- 延续当前统一 JSON 包装风格
- 返回结构要适合前端直接渲染列表与详情

### 任务 B6：注册路由并接入应用启动

修改文件：

- `apps/api-go/internal/router/router.go`
- `apps/api-go/internal/bootstrap/app.go`

要求：

1. 注入新的 PipelineService 依赖
2. 注册新的 pipeline routes
3. 不破坏现有 `/api/sessions`、`/api/tasks` 链路
4. 让系统可以同时支持旧 Demo 与新 Run 能力

### 任务 B7：建立 Session 到 PipelineRun 的桥接能力

修改文件：

- `apps/api-go/internal/service/publish_service.go`
- 或在 `pipeline_service.go` 中补充桥接逻辑

目标：

- 当前用户仍然可以先创建 Session
- 后续通过一个明确动作，把 Session 内容转为 PipelineRun
- 这样既保住当前 Demo 的对话入口，也开始迁移到新主模型

桥接策略：

- 读取 Session 标题与消息摘要
- 生成 `PipelineRun.requirementText`
- 记录 `SourceSessionID`
- 初始化默认模板与 StageRun

## 4.3 后端实施顺序

建议严格按以下顺序开发：

1. B1 模型
2. B2 Repository
3. B3 模板与阶段定义
4. B4 Service
5. B5 Controller
6. B6 Router / Bootstrap
7. B7 Session 桥接

这是为了避免先写接口，再反复回头改模型。

## 4.4 后端验收标准

完成后应能验证：

1. 数据库自动迁移成功
2. 默认模板已写入数据库
3. 可以通过 API 创建 PipelineRun
4. 创建时会自动生成 StageRun 记录
5. 可以通过 API 查询 Run 列表和详情
6. 详情中能看到 stages 与 artifacts 字段
7. 可以从现有 Session 成功创建一个 PipelineRun

## 5. 前端实施计划

## 5.1 前端阶段目标

前端第一阶段不追求做完整工作台，而是先完成 **新数据结构接入与基础页面承接**。

重点是：

- 接住后端新的 PipelineRun API
- 让现有页面不再只是静态原型
- 形成后续第二阶段继续扩展的前端骨架

## 5.2 前端具体任务拆分

### 任务 F1：建立 Pipeline API 调用层

建议新增文件：

- `apps/web/src/lib/api.ts`
- `apps/web/src/lib/pipeline.ts`

职责：

1. 封装 Pipeline API 请求
2. 定义前端使用的数据类型：
   - `PipelineRun`
   - `StageRun`
   - `Artifact`
   - `Checkpoint`
3. 统一处理 JSON 包装响应

第一阶段至少需要：

- `listPipelineRuns`
- `getPipelineRun`
- `createPipelineRun`
- `createPipelineRunFromSession`
- `listPipelineTemplates`（如果后端提供）

### 任务 F2：调整首页创建入口

修改文件：

- `apps/web/src/pages/Home.tsx`

第一阶段建议：

1. 保留当前创建 Session 的能力
2. 增加一个“直接创建 PipelineRun”入口，或在创建 Session 后提供转换入口
3. 首页输入需求后，允许两条路径：
   - 路径 A：进入 Session 做需求澄清
   - 路径 B：直接创建 PipelineRun

如果要更保守，可以第一阶段先只保留 Session 路径，但增加“从 Session 创建 PipelineRun”的按钮入口。

### 任务 F3：改造工作流页为真实数据页

修改文件：

- `apps/web/src/pages/Workflows.tsx`

当前该页已经有流程画布原型，非常适合承接第一阶段。

第一阶段改造目标：

1. 不要求立即支持拖拽编排保存
2. 先用于展示：
   - 默认模板阶段结构
   - 某个 PipelineRun 的阶段状态
3. 左侧或顶部增加 Run 列表/选择器
4. 点击某个 Run 后，右侧展示基本信息：
   - 标题
   - 状态
   - 当前阶段
   - 创建时间

### 任务 F4：为审批页和交付页接入新数据留接口

修改文件：

- `apps/web/src/pages/Approvals.tsx`
- `apps/web/src/pages/Delivery.tsx`

第一阶段不需要把这两个页面完全动态化，但要做两件事：

1. 抽离静态假数据
2. 改造成支持从统一 Pipeline 数据源读取 props / state

也就是先把它们从“纯展示原型”改成“可以挂真实数据的页面壳”。

### 任务 F5：整理应用路由与状态入口

修改文件：

- `apps/web/src/App.tsx`
- `apps/web/src/main.tsx`

目标：

1. 确保 Pipeline 页面路由清晰
2. 后续新增详情页时无需重构根结构
3. 为统一 API 层和后续状态管理留入口

如果当前 `App.tsx` 已经有基础路由结构，则第一阶段只做小规模整理，不做激进重构。

## 5.3 前端实施顺序

建议顺序：

1. F1 API 层
2. F2 首页入口
3. F3 Workflows 接真实数据
4. F4 Approvals / Delivery 留动态接口
5. F5 路由整理

## 5.4 前端验收标准

完成后应能验证：

1. 前端能请求 PipelineRun 列表
2. 前端能查看某个 PipelineRun 详情
3. Workflows 页面能展示真实阶段状态，而不是纯静态节点
4. 首页或 Session 页面可以触发创建 PipelineRun
5. Approvals / Delivery 页面结构已具备接真实数据的能力

## 8. 第一阶段执行原则

1. 不删除旧 Session 链路，先并存
2. 不过度抽象，优先把比赛主模型建立起来
3. 优先保证后端模型稳定，再推进前端接入
4. 第一阶段完成后，再进入第二阶段的 Pipeline 执行与审批流落地

---

## 9. 第一阶段完成后的下一步

第一阶段完成后，第二阶段将直接承接：

1. Pipeline 执行引擎
2. StageRun 实际推进
3. Checkpoint 审批流
4. Artifact 真实产出
5. 前端审批页与交付页动态化

因此第一阶段的核心价值不是“做出很多可见功能”，而是为后续赛题完整链路把地基打稳。
