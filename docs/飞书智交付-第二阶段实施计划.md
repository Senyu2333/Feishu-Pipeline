# 飞书智交付 · 第二阶段实施计划

## 1. 文档目标

本文档用于承接《飞书智交付赛题》《飞书智交付 · 产品开发设计》和《飞书智交付 · 第一阶段实施计划》，作为项目第二阶段后端开发的技术基线。

第一阶段已经完成 Pipeline 核心领域模型、默认模板、阶段运行、产物、检查点、基础 REST API、异步 Runner 和确定性 Pipeline 执行器。第二阶段的目标是在不推翻现有架构的前提下，把当前“Pipeline 数据底座 + Demo Executor”升级为更接近赛题要求的 **AI 驱动研发流程引擎（DevFlow Engine）可演示闭环**。

第二阶段优先围绕赛题的功能一展开：

> 用户输入自然语言需求，平台将其拆解为多阶段 Pipeline，每个阶段由 AI Agent 或确定性降级执行器完成，关键节点由人类审批，最终产出可验证、可评审、可交付的代码变更和交付摘要。

功能二“页面圈选元素 + 对话修改 + 热更新预览 + MR 提交”暂不作为第二阶段主线，但第二阶段需要继续保留 `InPageEditSession`、`GitDelivery` 等模型，为后续功能二接入留下扩展点。

### 2.2 当前代码的主要问题

第二阶段需要优先修正以下问题。

#### 2.2.1 PipelineRun 创建缺少事务

当前 `PipelineService.CreatePipelineRun` 按顺序创建：

1. `PipelineRun`
2. `StageRun`
3. `Checkpoint`
4. 初始 `Artifact`

这些数据库操作不是一个事务。如果中间步骤失败，可能留下半初始化数据。

影响：

- 前端可能看到不可执行的 PipelineRun
- Engine 可能遇到缺失阶段或缺失 checkpoint
- 本地演示稳定性下降

第二阶段应新增 repository 层事务方法。

#### 2.2.2 Pipeline 状态流转缺少合法性校验

当前 `StartPipelineRun`、`PausePipelineRun`、`ResumePipelineRun`、`TerminatePipelineRun` 主要直接更新状态，未严格限制状态转换。

风险示例：

- `completed` 后仍可能被 resume
- `terminated` 后仍可能被 start
- `waiting_approval` 可能被普通 resume 绕过审批
- 已终止 run 可能被重新入队

第二阶段应补充状态机校验。

#### 2.2.3 Checkpoint 缺少重复审批保护

当前 approve/reject 前未严格校验：

- checkpoint 是否仍是 `pending`
- 对应 StageRun 是否为 `waiting_approval`
- 对应 PipelineRun 是否为 `waiting_approval`
- checkpoint 是否已经被 approve/reject

第二阶段应补充审批前置校验，避免重复审批和非法推进。

#### 2.2.4 Stage 调度仍硬编码默认阶段

当前 `PreviousExecutableStage`、`NextStagesForReset`、`NextStage` 依赖默认 8 阶段硬编码。

这对当前默认模板可用，但不利于后续：

- 多模板
- 可配置模板
- 并行阶段
- 阶段依赖管理

第二阶段可以先保持顺序调度，但需要为模板化调度预留 `DependsOn` 等字段。

#### 2.2.5 `executor.go` 职责过重

当前一个文件中混合了：

- 阶段分发
- 需求分析产物生成
- 方案设计产物生成
- 代码变更计划生成
- 测试命令执行
- 代码评审报告生成
- 交付摘要生成
- JSON 提取工具

第二阶段接入真实 Agent Provider 后，该文件会继续膨胀。应拆分 Stage Handler。

#### 2.2.6 AgentRun 仍是确定性模拟记录

当前 Pipeline 执行时写入：

- `Provider: internal`
- `Model: deterministic`

这不是赛题要求中的真实 AI Agent 编排。第二阶段需要支持真实 Provider，同时保留 deterministic fallback。

#### 2.2.7 GitDelivery 模型已存在，但交付动作未实现

当前 delivery 阶段只生成交付摘要，没有真正实现：

- 创建工作分支
- 应用 patch
- commit
- PR/MR 摘要
- GitDelivery API

第二阶段应先实现 GitDelivery 基础记录和交付摘要 API，不默认执行 push 或创建远程 PR。

## 3. 第二阶段总体目标

第二阶段完成后，后端应支持以下完整演示链路：

1. 用户输入自然语言需求
2. 创建 PipelineRun
3. 后台执行需求分析阶段
4. 后台执行方案设计阶段
5. 到达方案审批 checkpoint，Pipeline 暂停
6. 人类 approve 后继续执行
7. 执行代码生成阶段，生成 diff/patch 计划或受控 patch
8. 执行测试阶段，生成测试报告
9. 执行代码评审阶段，生成评审报告
10. 到达评审确认 checkpoint，Pipeline 暂停
11. 人类 approve 后进入交付阶段
12. 生成交付摘要与 GitDelivery 记录
13. 前端可通过 API 展示完整阶段流、产物、测试报告、评审报告、审批状态和 AgentRun 记录

第二阶段不要求一次性实现完全自动提交 PR，但应完成“可验证交付摘要 + GitDelivery 记录 + 真实或降级 AI Agent 产物”的演示闭环。

## 5. 技术设计原则

第二阶段开发应遵循以下原则。

### 5.1 保留现有架构，不推翻重写

当前 `controller -> service -> repo -> pipeline` 分层是合理的。第二阶段应在该结构内演进。

推荐保持：

- HTTP 入参出参仍放在 `internal/type/pipeline`
- HTTP 控制器仍放在 `internal/controller`
- 业务编排仍放在 `internal/service`
- 数据访问仍放在 `internal/repo`
- Pipeline 引擎、调度、阶段处理仍放在 `internal/pipeline`

### 5.2 先保证稳定演示，再增强智能度

真实 AI Provider 可能受 API Key、网络、模型输出格式影响。第二阶段必须保留 deterministic fallback：

- 本地无 key 可以跑完整流程
- 有 key 可以开启真实 Agent 模式
- 真实 Agent 失败时应保存错误，而不是让数据丢失

### 5.3 所有阶段产物必须结构化

每个阶段的输出都应该是可解析 JSON，且同时保留可读文本。

Artifact 建议继续使用：

- `ContentText`：前端直接展示
- `ContentJSON`：下游阶段消费
- `MetaJSON`：状态、废弃、来源、版本等元信息

### 5.4 高风险动作默认不自动执行

以下行为默认不自动发生：

- `git push`
- 创建远程 PR/MR
- 删除文件
- 修改 `.env`、证书、密钥文件
- 大范围重写项目

如后续实现，需要显式配置或用户确认。

## 6. 详细实施计划

## 6.1 S2-1：Pipeline 状态机与事务一致性

### 6.1.1 目标

修正当前底座中最容易导致数据不一致的问题，使 PipelineRun 生命周期可靠。

### 6.1.2 需要修改的模块

主要文件：

- `apps/api-go/internal/service/pipeline_service.go`
- `apps/api-go/internal/repo/repository.go`
- `apps/api-go/internal/pipeline/stage.go`

建议新增文件：

- `apps/api-go/internal/pipeline/lifecycle.go`

### 6.1.3 技术方案

#### A. 新增事务创建方法

在 repository 层新增：

```go
func (r *Repository) CreatePipelineRunAggregate(
    ctx context.Context,
    run *model.PipelineRun,
    stages []model.StageRun,
    checkpoints []model.Checkpoint,
    artifacts []model.Artifact,
) error
```

内部使用 GORM transaction：

```go
return r.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error {
    if err := tx.Create(run).Error; err != nil {
        return err
    }
    if len(stages) > 0 {
        if err := tx.Create(&stages).Error; err != nil {
            return err
        }
    }
    if len(checkpoints) > 0 {
        if err := tx.Create(&checkpoints).Error; err != nil {
            return err
        }
    }
    if len(artifacts) > 0 {
        if err := tx.Create(&artifacts).Error; err != nil {
            return err
        }
    }
    return nil
})
```

`PipelineService.CreatePipelineRun` 只负责：

1. 读取模板
2. 解析模板
3. 组装 run/stages/checkpoints/artifacts
4. 调用 `CreatePipelineRunAggregate`
5. 查询详情返回

#### B. 新增状态机方法

在 `pipeline/lifecycle.go` 中定义：

```go
func CanStartRun(status model.PipelineRunStatus) bool
func CanPauseRun(status model.PipelineRunStatus) bool
func CanResumeRun(status model.PipelineRunStatus) bool
func CanTerminateRun(status model.PipelineRunStatus) bool
func CanApproveCheckpoint(run model.PipelineRun, stage model.StageRun, checkpoint model.Checkpoint) bool
func CanRejectCheckpoint(run model.PipelineRun, stage model.StageRun, checkpoint model.Checkpoint) bool
```

建议状态规则：

```text
draft -> queued
queued -> running
running -> waiting_approval
running -> failed
running -> completed
running -> paused
paused -> queued
failed -> queued
waiting_approval -> queued，只能由 approve/reject 触发
completed -> 终态
terminated -> 终态
```

#### C. 更新 service 方法

需要调整：

- `StartPipelineRun`
- `PausePipelineRun`
- `ResumePipelineRun`
- `TerminatePipelineRun`
- `ApproveCheckpoint`
- `RejectCheckpoint`

所有方法先读取当前 run/stage/checkpoint，再校验状态，再更新。

### 6.1.4 测试要求

新增或补充测试：

1. 创建 PipelineRun 成功时，run/stages/checkpoints/artifact 全部存在
2. 模拟中途失败时，不留下半初始化数据
3. `completed` 不能 resume
4. `terminated` 不能 start
5. `waiting_approval` 不能直接 resume 绕过 checkpoint
6. checkpoint 重复 approve 返回错误
7. checkpoint rejected 后不能再次 approve 同一轮

### 6.1.5 验收标准

- `go test ./...` 通过
- 非法状态转换返回明确错误
- PipelineRun 创建具备事务一致性
- 审批动作不可重复执行

## 6.2 S2-2：拆分 Stage Handler

### 6.2.1 目标

降低 `executor.go` 复杂度，为每个阶段接入真实 Agent、Prompt、Schema、测试做准备。

### 6.2.2 需要修改的模块

现有文件：

- `apps/api-go/internal/pipeline/executor.go`

建议新增：

- `apps/api-go/internal/pipeline/stage_handler.go`
- `apps/api-go/internal/pipeline/stage_requirement.go`
- `apps/api-go/internal/pipeline/stage_solution.go`
- `apps/api-go/internal/pipeline/stage_codegen.go`
- `apps/api-go/internal/pipeline/stage_test.go`
- `apps/api-go/internal/pipeline/stage_review.go`
- `apps/api-go/internal/pipeline/stage_delivery.go`
- `apps/api-go/internal/pipeline/stage_payload.go`

### 6.2.3 技术方案

定义统一 handler：

```go
type StageHandler interface {
    Execute(context.Context, StageContext) (StageExecutionResult, error)
}
```

`SequentialExecutor` 调整为持有 handler map：

```go
type SequentialExecutor struct {
    handlers map[string]StageHandler
}
```

初始化：

```go
func NewSequentialExecutor() *SequentialExecutor {
    return &SequentialExecutor{
        handlers: map[string]StageHandler{
            StageRequirementAnalysis: RequirementAnalysisHandler{},
            StageSolutionDesign:      SolutionDesignHandler{},
            StageCodeGeneration:      CodeGenerationHandler{},
            StageTestGeneration:      TestGenerationHandler{},
            StageCodeReview:          CodeReviewHandler{},
            StageDelivery:            DeliveryHandler{},
        },
    }
}
```

执行：

```go
func (e *SequentialExecutor) Execute(ctx context.Context, stageContext StageContext) (StageExecutionResult, error) {
    handler, ok := e.handlers[stageContext.Stage.StageKey]
    if !ok {
        return DefaultStageHandler{}.Execute(ctx, stageContext)
    }
    return handler.Execute(ctx, stageContext)
}
```

当前 deterministic 逻辑先平移，不改变行为。

### 6.2.4 验收标准

- 现有 pipeline engine 行为不变
- 原有测试通过
- 每个阶段 handler 可独立测试
- `executor.go` 只负责分发，不再承载所有阶段业务逻辑

```json
  "prTitle": "PR 标题",
  "prBody": "PR 描述",
  "manualReleaseNotes": ["人工发布说明"]
}
```

Artifact：

- `ArtifactType`: `delivery_summary`
- `Title`: `交付摘要`

### 6.3.9 JSON 解析失败策略

真实 LLM 输出可能不是合法 JSON。处理策略：

1. 保存 raw output 到 `AgentRun.OutputJSON` 或 `AgentRun.ErrorMessage`
2. 保存简短错误到 `StageRun.ErrorMessage`
3. 将 StageRun 标记为 `failed`
4. 将 PipelineRun 标记为 `failed`
5. 前端展示“模型输出格式错误，需要重试”

### 6.3.10 验收标准

- 每个阶段都有明确 schema
- 每个阶段产物可被下游消费
- JSON 解析失败不会丢失原始输出
- 前端可以根据 artifact type 渲染结构化详情

## 6.4 S2-4：Agent Provider Adapter

### 6.4.1 目标

把当前 deterministic executor 升级为可接真实模型的 Agent 执行体系，同时保持无 key 可演示。

### 6.4.2 建议新增模块

建议新增：

- `apps/api-go/internal/pipeline/provider.go`
- `apps/api-go/internal/pipeline/provider_deterministic.go`
- `apps/api-go/internal/pipeline/provider_ai.go`

也可以复用现有：

- `apps/api-go/internal/external/ai`

### 6.4.3 接口设计

```go
type GenerateRequest struct {
    StageKey      string
    SystemPrompt  string
    UserPrompt    string
    InputJSON     string
    Provider      string
    Model         string
    Temperature   float64
    MaxTokens     int
}

type GenerateResponse struct {
    Provider       string
    Model          string
    ContentText    string
    ContentJSON    string
    RawOutput      string
    TokenUsageJSON string
    LatencyMS      int64
}

type AgentProvider interface {
    Generate(context.Context, GenerateRequest) (GenerateResponse, error)
}
```

### 6.4.4 Provider 策略

第二阶段至少支持两类：

1. `deterministic`
   - 无需 API Key
   - 复用当前 deterministic 逻辑
   - 用于本地 demo 和测试

2. `ark` 或已有 AI client
   - 使用当前项目已有 AI 配置
   - 用于真实模型调用

后续可以扩展：

- Anthropic Claude
- OpenAI
- DeepSeek
- 通义

### 6.4.5 配置建议

在配置中增加：

```yaml
pipeline:
  mode: deterministic
  defaultProvider: deterministic
  defaultModel: deterministic
  enablePatchApply: false
```

真实 Agent 模式：

```yaml
pipeline:
  mode: agent
  defaultProvider: ark
  defaultModel: your-model-name
  enablePatchApply: false
```

### 6.4.6 AgentRun 写入要求

AgentRun 需要记录：

- provider
- model
- prompt snapshot
- input json
- output json
- token usage json
- latency ms
- status
- error message

当前已有字段基本足够，第二阶段需要保证真实填充。

### 6.4.7 验收标准

- deterministic 模式可完整跑通 Pipeline
- agent 模式至少能真实执行需求分析、方案设计、代码评审
- AgentRun 可查到真实 provider/model/latency/token 信息
- Provider 调用失败时 stage failed，错误可展示

## 6.6 S2-6：测试执行阶段增强

### 6.6.1 目标

让测试阶段成为真实质量门禁，而不是只输出文字说明。

### 6.6.2 当前状态

当前当 `TargetRepo == "self"` 时，会运行：

```bash
go test ./internal/pipeline ./internal/service
```

第二阶段建议扩展为更通用的测试执行器。

### 6.6.3 技术方案

新增：

- `apps/api-go/internal/pipeline/test_runner.go`

接口：

```go
type TestCommandResult struct {
    Command    string `json:"command"`
    ExitCode   int    `json:"exitCode"`
    Status     string `json:"status"`
    Stdout     string `json:"stdout"`
    Stderr     string `json:"stderr"`
    DurationMS int64  `json:"durationMs"`
}

type TestRunner interface {
    Run(context.Context, model.PipelineRun, []string) ([]TestCommandResult, error)
}
```

当前项目默认命令：

```bash
go test ./...
```

后续前端接入后可增加：

```bash
npm run build
npm test
```

### 6.6.4 失败策略

第二阶段建议：

- 测试命令失败时，测试阶段仍生成 `test_report`
- StageRun 可以标记 `succeeded`，但报告状态为 `failed`
- 后续 `code_review` 阶段读取测试失败并给出风险结论

原因：

- 比赛演示中更容易展示完整链路
- Review Agent 可以解释失败原因
- 人类在 checkpoint 决定是否继续

如要更严格，可以配置：

```yaml
pipeline:
  failOnTestFailure: false
```

### 6.6.5 验收标准

- 测试报告包含 command、exitCode、stdout、stderr、duration
- 测试失败不丢日志
- Review 阶段能读取测试结果

## 6.7 S2-7：Review Agent 与修复闭环雏形

### 6.7.1 目标

让代码评审阶段真正基于需求、方案、diff、测试报告做判断。

### 6.7.2 技术方案

Review 阶段输入：

- `structured_requirement`
- `solution_design`
- `code_diff`
- `test_report`

Review 输出：

```json
{
  "conclusion": "approved|needs_fix|blocked",
  "issues": [],
  "securityNotes": [],
  "maintainabilityNotes": [],
  "testCoverageNotes": []
}
```

### 6.7.3 与 Checkpoint 的关系

Review 阶段结束后进入 `checkpoint_review`。

前端审批页应展示：

- diff 摘要
- 测试报告
- review conclusion
- issues
- security notes

用户操作：

- approve：进入 delivery
- reject：回退到 `code_review` 或 `code_generation`

当前代码中 `PreviousExecutableStage(StageCheckpointReview)` 返回 `StageCodeReview`，第二阶段可以先保持该逻辑。

### 6.7.4 自动修复策略

第二阶段只做雏形，不做完全自动循环。

建议提供：

```text
POST /api/pipeline-runs/:id/stages/:stageKey/retry
```

或继续通过 reject checkpoint 携带理由回退。

### 6.7.5 验收标准

- Review report 能展示具体问题和建议
- 测试失败时 review conclusion 不应无脑 approved
- reject reason 能进入下一次 Stage input

## 6.8 S2-8：受控 Patch / Diff 产物

### 6.8.1 目标

让代码生成阶段从“变更计划”升级为“可验证 diff/patch 产物”。

第二阶段不要求默认自动修改所有文件，但应具备受控 patch 能力。

### 6.8.2 建议新增模块

- `apps/api-go/internal/pipeline/patch.go`
- `apps/api-go/internal/pipeline/workspace.go`

### 6.8.3 Patch 数据结构

```go
type FilePatch struct {
    FilePath   string `json:"filePath"`
    ChangeType string `json:"changeType"`
    Patch      string `json:"patch"`
    Reason     string `json:"reason"`
}
```

### 6.8.4 安全规则

Patch 应用前必须检查：

1. 文件路径必须在项目根目录内
2. 禁止修改 `.env`、`.pem`、`.key`、`.crt`、`credentials` 等敏感文件
3. 禁止删除目录
4. 限制单文件大小
5. 限制一次变更文件数量
6. 默认不 commit
7. 默认不 push

### 6.8.5 执行模式

建议分两种模式：

```yaml
pipeline:
  enablePatchApply: false
```

- `false`：只生成 patch artifact，不应用到工作区
- `true`：在安全检查通过后应用 patch

第二阶段建议默认 `false`，比赛演示时可以对 demo 文件开启。

### 6.8.6 验收标准

- CodeGeneration 阶段可以输出 patch 列表
- Patch artifact 可以被前端展示
- 安全检查失败时给出明确错误
- 不会默认修改敏感文件

## 8. 第二阶段最终验收清单

第二阶段完成后，应满足以下验收标准。

### 8.1 Pipeline 创建与执行

- 可以创建 PipelineRun
- 自动初始化 8 个 StageRun
- 自动初始化两个 Checkpoint
- 自动生成初始需求 Artifact
- 创建过程具备事务一致性
- 后台 Runner 能执行 PipelineRun

### 8.2 状态机

- 非法状态转换会失败
- completed/terminated 是终态
- waiting_approval 只能通过 checkpoint 操作继续
- checkpoint 不能重复审批

### 8.3 阶段产物

每个可执行阶段均有结构化产物：

- `structured_requirement`
- `solution_design`
- `code_diff`
- `test_report`
- `review_report`
- `delivery_summary`

### 8.4 AgentRun

- 每个可执行阶段都有 AgentRun
- AgentRun 记录 provider、model、input、output、status、latency
- deterministic 模式和 agent 模式都可运行

### 8.5 Checkpoint

- 方案审批能暂停和继续
- 评审确认能暂停和继续
- reject 能回退上一阶段
- reject reason 能进入下一轮阶段输入

### 8.6 测试与评审

- 测试阶段保存命令结果
- 测试失败时 review 能识别风险
- review report 有结论和问题列表

### 8.7 交付

- Delivery 阶段生成交付摘要
- 可以创建或查询 GitDelivery 记录
- 默认不 push、不创建远程 PR

### 8.8 前端 API

- timeline API 可渲染阶段流
- current API 可渲染当前阶段详情
- checkpoint 页面可展示审批所需数据
